
# 修复：保存分析结果后出现重复问题（英文 + 中文各 8 个）

## 根本原因

`Analytics.tsx` 第 88-89 行的 `handleSaveAnalysis` 函数在保存问题时使用了数组追加（spread）：

```ts
// 当前的错误逻辑：追加而非替换
questions: [...(s.questions || []), ...questions],
```

这意味着每次保存分析结果时，新的问题会**叠加**到旧的问题上，而不是替换。具体场景：

- 场景 1：stage 里已有英文的旧问题（比如 demo 数据），中文新问题被追加进去 → 英文 8 个 + 中文 8 个 = 16 个
- 场景 2：用户重新分析了两次并保存 → 第一次的问题 + 第二次的问题叠加 = 两倍

## 修复方案

### 改动 1：`src/pages/Analytics.tsx` — 保存时替换而非追加

将第 88 行的追加改为直接替换：

```ts
// 修复后：替换而非追加
questions: questions,
```

这样每次保存分析结果，都会以最新的分析结果**完整替换**旧的问题列表，不会产生重复。

### 改动 2（同一函数）：同时也需要处理 pipelines 里的 stages

当前 `handleSaveAnalysis` 只更新了 `job.stages`，但数据实际存储结构里还有 `job.pipelines[].stages`。需要确认是否需要同步更新 pipelines 里相同 stageId 的问题（检查数据流后决定）。

## 额外清理（可选但建议）

对于已经保存了重复问题的现有数据，用户可以通过"重新分析 → 保存"来覆盖，修复后就不会再追加了。不需要特殊迁移脚本。

## 需要修改的文件

| 文件 | 修改内容 |
|---|---|
| `src/pages/Analytics.tsx` | `handleSaveAnalysis` 中第 88 行，将 `[...(s.questions || []), ...questions]` 改为 `questions` |

## 改动范围

- 极小改动，只改一行
- 不涉及数据库 schema
- 不影响其他功能
- 修复后旧的重复数据需用户重新分析一次以覆盖（不自动清理历史重复数据）
